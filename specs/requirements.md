# Requirements — Млечный Путь Web App (Frontend)

## 1. Контекст

Логистическая компания «Млечный Путь» (mleko.kz, основана в 2013). Есть готовый
HTML-макет (`Млечный Путь - Веб-макет (автономный).html`) — презентация из 13
экранов будущего сайта. Нужно превратить макет в работающее React-приложение.

## 2. Цель этой итерации

Сделать **верстку** (React + TypeScript + Tailwind CSS) всех экранов из макета:
живой, кликабельный фронтенд с мок-данными, без реального бэкенда. Архитектура
данных должна быть готова к замене мок-слоя на вызовы FastAPI-бэкенда позже,
без переписывания компонентов.

Вне скоупа этой итерации:
- Реальный бэкенд (FastAPI) — добавится позже, в `backend/`.
- Реальная аутентификация (личный кабинет работает на мок-пользователе).
- Реальная отправка форм (e-mail/WhatsApp) — кнопки работают, но шлют в мок-API.
- Полный i18n (EN/TR) — UI переключателя языка есть, но контент только на RU.

## 3. Функциональные требования

### 3.1 Публичный сайт
- FR-1: Главная страница с hero, переключаемой каруселью «горячих предложений»
  (autoplay + ручное управление), блоком преимуществ/контактов, футером.
- FR-2: Каталог услуг с поиском, сортировкой, фильтрами по категориям и сеткой
  карточек услуг (цена «от», кнопка расчёта).
- FR-3: Страница «Международные перевозки» — дерево направлений по видам
  транспорта (Авто/Авиа/Море/ЖД), бегущая лента маршрутов, этапы процесса, CTA.
- FR-4: Страница «Внутри Казахстана» — авиа/авто перевозки внутри страны,
  дверная доставка по городам, плашка «бронирование в тестовом режиме».
- FR-5: Страница «Доп. услуги» — упаковка/склад/брокер/животные/скоропорт/
  спецтехника/опасные грузы (9 классов)/грузчики/дверная доставка.
- FR-6: Форма расчёта стоимости — выбор вида транспорта, маршрут, параметры
  груза, контакты, способ ответа; отправка через мок-API; боковая панель
  «как это работает» + контакты.
- FR-7: Страница бронирования (`/bron/:id`) — статус-степпер, детали перевозки,
  выставление счёта (выбор способа оплаты), таблица «мои бронирования».
- FR-8: Все публичные страницы — переключатель языка (RU активен, EN/TR в UI),
  кнопка WhatsApp (плавающая), единые nav/footer.

### 3.2 Личный кабинет
- FR-9: Layout с сайдбаром (профиль, навигация с бейджами, выход) и топбаром
  (поиск, кнопка «Новая заявка», уведомления, аватар).
- FR-10: Дашборд заявок — карточки статистики, таблица заявок с фильтрами по
  статусу, выбор строки.
- FR-11: Панель деталей груза (справа) — визуализация маршрута с прогрессом,
  таймлайн статусов отправки, детали груза, список документов, действия.

### 3.3 Адаптивность
- FR-12: Все страницы должны корректно работать на desktop (≥1280px), tablet
  (~768px) и mobile (375px) — а не только в виде 3 готовых мобильных мокетов
  из макета (Главная/Расчёт/Бронирование).

## 4. Нефункциональные требования

- NFR-1: Tailwind CSS — единственный механизм стилизации (никаких отдельных
  CSS-модулей с дублирующей логикой токенов).
- NFR-2: Цвета/шрифты/радиусы вынесены в Tailwind theme как токены дизайн-
  системы из макета (см. design.md) — не хардкодить hex в компонентах.
- NFR-3: Данные (услуги, направления, цены, бронирования, заявки кабинета)
  лежат в отдельном слое `src/api/` с интерфейсами TS — сейчас отдаёт мок-JSON,
  в будущем заменяется на `fetch` к FastAPI без изменения компонентов.
- NFR-4: Компонентная структура atomic-design (atoms → molecules → organisms →
  pages), переиспользуемые компоненты не дублируются между страницами.
- NFR-5: Роутинг — React Router, маршруты соответствуют разделу 6 design.md.
- NFR-6: Текстовый контент берётся из макета как есть (копирайтинг уже готов).

## 5. Критерий готовности (Definition of Done)

- Все 8 публичных маршрутов + кабинет открываются и визуально соответствуют
  макету (цвета, типографика, сетки, состояния).
- Приложение собирается (`npm run build`) без ошибок типов/линта.
- Адаптивность проверена на 3 брейкпоинтах вручную в браузере.
- Мок-данные централизованы и типизированы, заменяемы на реальный API.
