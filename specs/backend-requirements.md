# Requirements — Backend (итерация 2: Бронирование + Личный кабинет)

## 1. Контекст

Фронтенд (см. `requirements.md`/`design.md`) уже сверстан на мок-данных за
слоем `frontend/src/api/*Api.ts`. Теперь подключаем настоящий бэкенд для двух
разделов, которые реально нужны прямо сейчас: **бронирование груза**
(`/bron/:id`) и **личный кабинет** (`/cabinet/*`). Остальные разделы (каталог,
международные, расчёт стоимости и т.д.) остаются на моках до следующей
итерации — расширяются по той же схеме, когда понадобится.

## 2. Скоуп

В скоупе:
- FastAPI-приложение в `backend/`, REST API, JSON.
- MongoDB как хранилище (коллекции `bookings`, `requests`, `users`).
- Эндпоинты для бронирований и личного кабинета (см. design.md §3).
- Seed-скрипт, заполняющий Mongo данными из `specs/content-inventory.md`
  (те же бронирования/заявки, что сейчас в моках фронтенда) — чтобы переход
  с мока на реальный API был незаметен визуально.
- Замена тел функций в `frontend/src/api/bookingsApi.ts` и `cabinetApi.ts` на
  `fetch` к новому API (без изменения сигнатур и без изменения компонентов —
  именно для этого и был выделен слой `*Api.ts` в итерации 1).
- CORS, настроенный на dev-сервер фронтенда.
- `docker-compose.yml` с MongoDB для локальной разработки.

Вне скоупа (осознанно отложено):
- Аутентификация/JWT — кабинет работает с одним мок-пользователем без логина.
- Каталог услуг, расчёт стоимости, международные/КЗ-страницы — остаются на
  фронтенд-моках, бэкенд их не обслуживает.
- Создание новой заявки/бронирования через форму (кнопка «Новая заявка» в
  кабинете остаётся декоративной) — только чтение существующих + действие
  «выставить счёт».
- Realtime/WebSocket для трекинга — статус читается обычным GET.

## 3. Функциональные требования

- FR-1: Список бронирований пользователя (для таблицы «Мои бронирования» и
  GET `/api/bookings`).
- FR-2: Деталь бронирования по id (для `/bron/:id`).
- FR-3: Выставление счёта по бронированию — выбор способа оплаты
  (`requisites` | `remote`), сохраняется в Mongo, статус не меняется
  автоматически (как и в макете — это ручной шаг менеджера).
- FR-4: Данные кабинета: текущий пользователь, статистические карточки,
  список заявок с фильтрами по статусу, деталь заявки (маршрут+прогресс,
  таймлайн, груз, документы).
- FR-5: Статистика кабинета (`CabinetStats`) вычисляется на бэкенде из
  коллекции `requests` агрегацией — не хранится отдельно (DRY: один источник
  истины, нет риска рассинхронизации).

## 4. Нефункциональные требования

- NFR-1: **KISS** — без слоёв repository/service/DI-контейнеров сверх
  необходимого; роутер → Mongo-запрос → Pydantic-модель ответа.
- NFR-2: **DRY** — одна Pydantic-модель документа на коллекцию; ответы для
  списка/деталей строятся из неё через `model_dump(include=...)`, а не через
  отдельные дублирующиеся схемы с переписанными полями.
- NFR-3: Контракты ответов API совпадают по форме с TS-типами в
  `frontend/src/api/types.ts` (Booking, BookingDetail, CabinetUser,
  CabinetStat, CabinetRequest, CabinetRequestDetail) — это уже согласованный
  контракт из итерации 1, бэкенд под него подстраивается, а не наоборот.
- NFR-4: Без аутентификации — но код пишется так, что добавление auth-мидлвара
  позже не требует переписывания роутеров (user id не хардкодится внутри
  бизнес-логики, а лежит в одном месте — `config.py`/seed).

## 5. Definition of Done

- `docker compose up -d` поднимает MongoDB локально.
- `python -m app.seed` заполняет базу данными из content-inventory.md.
- `uvicorn app.main:app --reload` поднимает API на `:8000`.
- Фронтенд (`npm run dev`) показывает `/bron/MP-024815` и `/cabinet` с теми же
  данными, что раньше были в моках — но теперь они приходят через `fetch` из
  MongoDB.
- `GET /api/bookings`, `GET /api/bookings/{id}`, `POST /api/bookings/{id}/invoice`,
  `GET /api/cabinet/user`, `GET /api/cabinet/stats`, `GET /api/cabinet/requests`,
  `GET /api/cabinet/requests/{id}` отвечают 200 с корректной формой JSON.
